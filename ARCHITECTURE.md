# Простая архитектура MAX-бота ЖКХ на Go

## Назначение

Сервис является промежуточным backend-слоем между MAX и 1С. Он не публикует 1С напрямую в интернет, принимает Webhook-события от MAX, проверяет секрет, маршрутизирует пользовательские сценарии и вызывает HTTP API 1С.

## Компоненты

```text
Житель в MAX
   │
   ▼
MAX Bot API
   │ Webhook HTTPS /webhook/max
   ▼
Go backend
   ├─ httpserver: HTTP endpoint, проверка secret, healthcheck
   ├─ service: сценарии бота и бизнес-оркестрация
   ├─ store: идемпотентность событий и сессии диалогов
   ├─ clients/max: отправка сообщений в MAX
   └─ clients/onec: вызовы HTTP API 1С
   │
   ▼
1С HTTP API
   ├─ пользователи MAX и согласия
   ├─ привязка лицевых счетов
   ├─ баланс, приборы учета, показания
   └─ обращения и заявки
```

## Поток Webhook

1. MAX отправляет событие на `POST /webhook/max`.
2. Backend проверяет заголовок с секретом.
3. Backend быстро отвечает `HTTP 200 {"success":true}`.
4. Событие обрабатывается асинхронно.
5. Store выполняет `ClaimEvent` по `message_id` или вычисленному `event_id`, чтобы повтор события не создал дубль.
6. BotService вызывает 1С и отправляет пользователю ответ через MAX API.
7. Результат обработки фиксируется в Store.

## Основные сценарии

- `/start` — создает или обновляет пользователя MAX в 1С.
- `согласен` — фиксирует согласие на обработку данных в 1С.
- `привязать <номер ЛС>` — начинает привязку лицевого счета.
- `код <номер ЛС> <код>` — подтверждает привязку.
- `баланс` — получает активный лицевой счет и баланс из 1С.
- `показания` — получает список активных ИПУ.
- `показание <meter_id> <значение>` — передает показание в 1С.
- `обращение <текст>` — создает обращение по активному ЛС.
- `справка` — получает справочную информацию из 1С.

## Безопасность

- MAX token, Webhook secret, внутренний токен 1С и служебный токен уведомлений хранятся только в переменных окружения.
- Секрет Webhook проверяется константным сравнением.
- Секрет в URL не поддерживается.
- Публичный endpoint ведет только на Go backend, а не напрямую на 1С.
- В логах не пишутся токены и полный состав персональных данных.
- Адрес и серийный номер счетчика в ответах пользователю маскируются в примере.

## Ограничения каркаса

- Store сделан in-memory для простоты. Для production заменить на PostgreSQL/Redis с TTL и уникальными индексами.
- Асинхронная обработка сделана через goroutine. Для production заменить на очередь: RabbitMQ, NATS, Kafka, PostgreSQL outbox.
- Формат отправки сообщения в MAX вынесен в `clients/max/client.go`; его нужно сверить с фактической версией MAX Bot API.
- Сценарии диалога упрощены и построены на текстовых командах. Кнопки callback можно добавить в `model.MAXUpdate` и `BotService.handle`.
