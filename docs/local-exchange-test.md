# Локальный тест обмена MAX ↔ Go bot ↔ mock 1C

Этот контур проверяет бота без опубликованной 1С, без реального `ONEC_BASE_URL` и без подключения к базе 1С. В тестах используется `httptest` HTTP-сервер, который одновременно играет роль MAX mock (`/messages`) и ONEC mock (`/max/v1/...`).

## Что проверяется

- пользовательский сценарий `/start` → авторизация → ввод ЛС `000123456` → код `1234` → авторизованные действия;
- кнопки гостевого и авторизованного меню, включая непустые callback payload;
- обработка payload: `authorize`, `balance`, `meters`, `invoice`, `payment`, `appeal_start`, `outages`, `appointment`, `organization`, `emergency`, `help`;
- что запросы бота доходят до ONEC mock с ожидаемыми method/path/query/body/headers;
- `Authorization: Bearer TEST_ONEC_TOKEN` для каждого запроса к mock 1C;
- `max_user_id`, активный `account_id`, `source=MAX`, `message_id` и `operation_id` для изменяющих операций;
- ответы, отправленные ботом в MAX mock, по ключевым фразам, а не полным текстовым совпадениям;
- статическая сверка контракта Go-клиента с `cf_billing`.

## Mock-only данные

Тестовые значения явно отделены от реальных данных: `MOCK-ACC-000123456`, `MOCK-HVS-001`, `MOCK-GVS-001`, `MOCK-РП-000001`, `MOCK-КВ-000001`, `MOCK-APT-001`, `MOCK-APL-001`. Пользовательский ввод ЛС остается похожим на реальный (`000123456`), но mock 1C возвращает `MOCK-ACC-000123456`.

## Как запустить

```bash
go test ./...
./scripts/test-exchange-local.sh
```

PowerShell:

```powershell
./scripts/test-exchange-local.ps1
```

Скрипты не требуют опубликованной 1С. Docker не обязателен: если он найден, скрипт только сообщает, что основной контур уже покрыт `httptest`.

## Как читать отчет

Успешный запуск показывает:

- список callback payload, проверенных через кнопки;
- список ONEC endpoint, до которых дошли запросы;
- список ключевых MAX-сообщений, найденных в исходящих ответах бота;
- результат `TestOneCContractMatchesCfBilling` со списком `MISSING_IN_1C_CONFIG` в verbose-логе.

`MISSING_IN_1C_CONFIG` означает, что Go-клиент уже содержит endpoint, но в текущем dump `cf_billing` нет полного опубликованного HTTP handler + вызова `MAXBotИнтеграция` + экспортной функции. Это отчет о контракте, а не попытка изменить бизнес-логику 1С.

## Чем mock-сценарий отличается от реальной 1С

Mock не выполняет код 1С, не пишет в документы/справочники 1С, не отправляет SMS и не формирует реальные квитанции или платежные ссылки. Он проверяет локальный контракт бота: маршруты, методы, headers, JSON body/query и реакцию сервиса на APIResponse. Для реального сервера 1С после публикации HTTP-сервисов нужно отдельно выполнить smoke/integration тесты с настоящим `ONEC_BASE_URL` и тестовой информационной базой.
