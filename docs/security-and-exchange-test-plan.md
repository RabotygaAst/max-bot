# План проверки безопасности и обмена MAX Bot ↔ 1С

## Авторизация
- Запрос без `Authorization` → `401`.
- Запрос с неправильным `Bearer token` → `401`.
- Запрос с корректным токеном → `200`.

## Проверка доступа к ЛС
- `max_user_id A` запрашивает свой `account_id` → `200`.
- `max_user_id A` запрашивает чужой `account_id` → `403`.
- `max_user_id` без привязки запрашивает `balance`/`meters`/`invoice` → `403` или `AUTH_REQUIRED`.

## Защита от утечки приборов
- `GET meters` должен вернуть только приборы текущего ЛС.
- Ответ не должен содержать приборы другого абонента.
- Каждый `meter_id` из ответа должен принадлежать текущему ЛС.

## Показания
- Передать показание по своему прибору → `200` или бизнес-ошибка 1С.
- Передать показание по чужому `meter_id` → `403/404`.
- Передать значение меньше предыдущего → business error.
- Повторить тот же `operation_id` → не создавать дубль документа/записи.

## Начисления и услуги
- `balance` должен возвращать долг/переплату по ЛС.
- Детализация должна находить услуги/тарифы/начисления в базе 1С.
- Список услуг не должен быть пустым, если в базе есть начисления по услугам.

## SQL/BSL injection и некорректные параметры
- `account_id` с кавычками.
- `account_id` с пробелами.
- `meter_id` с кавычками.
- `period` не в формате `YYYY-MM`.
- `max_user_id` пустой.
- `max_user_id` строкой.

## Логи и ПДн
- В логах не должно быть токена.
- В логах не должно быть кода подтверждения.
- В логах не должно быть полного адреса, если это не технический журнал 1С с ограниченным доступом.
